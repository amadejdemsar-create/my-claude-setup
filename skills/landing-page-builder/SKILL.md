# Landing Page Builder

Build landing pages at the level the best agency on earth would ship: a team with 25 years each in brand strategy, conversion copywriting, art direction, UX, motion, front-end engineering, and CRO, working with obsessive, Tesla-grade attention to the last 2% that nobody else bothers with. The bar is not "good for an AI" or "good enough to deploy." The bar is: would a world-class studio put their name on this and charge $40k for it? If not, it is not done.

This skill is the **studio lead**. It owns the strategy, the narrative, and the conversion mechanics, and it conducts a roster of specialists for the parts they each do best. The point of the agency framing is not flavour: each discipline is a distinct expert **lens**, and running the work through all of them, in sequence and again as a critique, is what catches the flaws a single pass leaves in.

## The roster (your specialists)

Read `references/agency-roles.md` once at the start of a build for what each discipline demands and how to channel it. In short:

- **`impeccable` skill** — the principal engineer + art director. The build engine. Produces the production code and elevates craft. Invoke in Phase 4.
- **`design-taste-frontend` skill** — the creative director's anti-slop conscience and the pre-ship quality gate. Direction system in Phase 1/3, pre-flight checklist before shipping.
- **`ui-ux-pro-max` skill** — the design researcher's reference library. Query for the palette, font pairing, and style that fit this product type and audience before committing direction.
- **`emil-design-eng` skill** — the motion designer. Invoke for any real motion (hero entrance, scroll reveals, micro-interactions) so movement is purposeful and perfectly eased.
- **`copywriting-storytelling` agent** — the copy chief. Delegate the copy deck for complex offers or when the user wants multiple hook variants; otherwise write copy here against `references/copywriting-playbook.md`.
- **`design-visionary` agent** — the executive creative director. The deeper critique during QA when the user wants to push further.

Read these reference files at the phases noted; do not try to hold all of it in your head at once:
- `references/discovery-brief.md` — the consultant's intake: what to extract vs. infer, and how to run the discovery (Phase 0).
- `references/agency-roles.md` — the disciplines and the multi-lens critique (Phase 0, and the critique in Phase 5).
- `references/concept-boards.md` — the mockup-first iteration method: live switchable variant boards for the hero, then each section, before the full build (Phases 1 and 3).
- `references/copywriting-playbook.md` — world-class conversion copy (Phase 2).
- `references/design-principles.md` — structure + conversion layout rules (Phases 1, 3, 4).
- `references/craft-standards.md` — the obsessive "last 2%" detail bar (Phases 4 and 5).
- `references/conversion-checklist.md` — the QA gate (Phase 5).

## Toolchain (non-negotiable)

- **Images are never generated by an API, and never by Google Gemini.** Whenever the page or a concept needs generated imagery (hero art, backgrounds, illustration, social/OG image), do NOT call an image model. Invoke the **`image-director`** skill, which authors the prompt and renders it at quality:high via the Codex built-in image_gen tool (built-in only, never an API, never Gemini); drop the render in. Use this skill's direction + craft references to write a strong prompt; only the render step is external. Build the page with a labeled placeholder (correct dimensions, honest) where the generated image will go.
- **Real brand assets** (logos, provided photography) are used directly from the project's assets folder (e.g. `<root>/Business/<Company>/Assets/`), never hand-rolled or AI-generated. Stock imagery may use the user's curated sources; otherwise search via Firecrawl.
- **Any web search / fetch** (inspiration, competitors, stock, fonts) uses the **Firecrawl CLI**, not `curl`/`wget`.

## When to use this skill

- A new landing page (product, service, offer, app, agency, event).
- A lead-gen, sales, waitlist, beta-signup, webinar, or download page.
- A rework or iteration of an existing landing page (paste, file path, or URL).
- A high-fidelity page mockup to deploy or hand to a developer.

Do NOT use it for: full multi-page sites or web apps (route to a project build), blog posts, internal dashboards, or pages where conversion is irrelevant. For those, use `impeccable` directly (or `ui-ux-pro-max` for a dashboard's chart/data patterns) or a project codebase.

## The quality bar (non-negotiable)

A page passes only if every one of these is true. These are the floor, not the ceiling.

1. **One hook, one promise.** The visitor knows what this is, who it is for, and why to care within 5 seconds. No clever-but-vague headlines.
2. **A single, sharp strategic idea** runs through the whole page: one positioning, one core message, one primary action. Everything ladders to it.
3. **Visual hierarchy is inevitable.** The eye lands on headline → subhead → primary CTA, in that order, with zero effort.
4. **It has a point of view.** A committed aesthetic, distinctive type, intentional color, real atmosphere. Never Inter-on-white with a purple gradient and three identical cards.
5. **The craft survives a loupe.** Consistent spacing scale, optical alignment, tuned headline tracking, no orphans/widows, real hover/active/focus states, tinted neutrals not flat grey. The last 2% is done (`references/craft-standards.md`).
6. **Friction is minimized.** The primary action is obvious, repeated, and cheap to take. Forms ask for the minimum.
7. **It earns trust.** Specific social proof, real numbers, named outcomes, recognizable logos or testimonials, risk reversal. Never fabricated.
8. **Mobile is designed, not squashed.** Most traffic is mobile; the 390-wide view is art-directed first.
9. **It performs.** No layout shift, no FOIT, deliberate above-the-fold paint, accessible (WCAG AA).

If the inputs cannot support this bar, say so and ask for what is missing rather than shipping filler. World-class work refuses to fake the parts it cannot earn.

## Workflow

Seven phases. Hard confirmation gates at Phase 1 (direction lock) and Phase 3 (structure + copy lock) before building, and Phase 5 must pass before "done." Other boundaries are fluid; never silently skip a phase, and if you compress one, say which and why (`completion-discipline.md`).

### Phase 0: The discovery call (run it like a consultant)

You are the expert in the room. A great agency does not hand the client a 20-field form; it runs a tight discovery conversation, because it already knows exactly what it needs to produce great work and what it can decide itself. Behave that way. **Read `references/discovery-brief.md` now** for the full intake model; the essence:

1. **Mine context first, ask second.** Before asking anything, load what you already have (local brand files below, any URL or page the user named, the conversation so far). Never ask for something you can read or reasonably infer. Every avoidable question costs you credibility as the expert.
2. **Know the difference between must-have and inferable.** A handful of inputs only the user can give, and getting them wrong makes the output mediocre or fake: the **real offer**, the **one action** + where it goes (form destination/backend), the **actual proof** (real numbers, names, logos you may use), **brand truth** (voice, assets, hard constraints), and **which project** this belongs to (sets the form-notification email identity + file location; never guess which project owns the page). Everything else (positioning angle, awareness stage, structure, aesthetic direction, palette, copy) you **propose** as an informed default and let the user correct.
3. **Ask the fewest, sharpest questions, in one batched round.** Use `AskUserQuestion` for the structured choices (project, goal, output format), prose for the open ones (offer, proof). Lead with the highest-leverage unknown. Pre-fill smart defaults so the user is editing a strong proposal, not facing a blank form. Aim for one round; do not drip-feed.
4. **Name the riskiest unknown.** State the single input that most determines whether this page wins (usually: is there real, specific proof, or a sharp differentiator). Make sure it is nailed before building.
5. **Mirror the brief back and lock it.** Before Phase 1, restate a one-paragraph brief plus the assumptions you are making on the inferable items, and ask the user to confirm or correct. This is the moment a consultant earns trust and prevents three rebuilds.
6. **Refuse to fake.** If a must-have is missing (no real proof, no offer clarity, no destination), say so plainly and either get it or design around the gap honestly. World-class work never invents proof to fill a hole.

Load local context first (CLAUDE.md "local first"):
- A business with an established brand: its `Context/` + `Assets/` folders (use a brand agent or brand tokens if the project has one).
- Personal project: its own `Context/` folder.
- No brand yet: say so, propose a clean direction in Phase 1, never invent a logo (use a wordmark in the brand font).

Also fix the **output format**: self-contained Tailwind HTML (default) or a Next.js App Router page (only if the target is an existing Next.js codebase).

If iterating on an existing page: read it first (file, paste, or fetched URL), diagnose against the quality bar before proposing changes, and preserve what works rather than rewriting wholesale. Then your questions are sharper: ask only about the specific weaknesses, not the whole brief.

### Phase 1: Strategy + visual direction (mandatory, never skip)

No copy, no design, no code until this is locked. World-class pages start from a sharp strategic idea and a committed look, not from riffing blind.

1. **Set the strategy** (the marketing strategist's lens, see `references/copywriting-playbook.md` "Strategy"). Pin down: the **positioning** (what category this competes in and why it wins), the visitor's **awareness stage** (unaware → most aware; this dictates how much you explain vs. how hard you close), the **single core message**, the **primary objection** to disarm, and the **one action**. Write these as one tight block.
2. **Read `references/design-principles.md`** for structure and visual rules.
3. **Gather references.** 4 to 8 concrete visual references for this offer/audience. In order: any inspiration URLs the user gave (scrape with Firecrawl, formats `["html","markdown","screenshot"]`; for logged-in/anti-bot, see `~/.claude/rules-ondemand/web-operations.md`); the user's own site/assets in your workspace; a targeted Firecrawl search for best-in-class pages in the category (verify any "current best"/trend claim per `~/.claude/rules/fast-moving-identifiers.md`, do not assert from memory); Mobbin for real product UI if design-led.
4. **Commit to a direction.** Channel `design-taste-frontend`'s anti-slop philosophy and query `ui-ux-pro-max` for the palette + font pairing that fit this product type and audience. Pick ONE aesthetic point of view (refined minimal, editorial, bold maximalist, brutalist, luxury, retro-futuristic, organic, technical/utilitarian, etc.). Decide: dominant color + accent, display font + body font (distinctive, not Inter/Roboto/Arial unless the brand mandates), the one memorable element, light vs dark, density.
5. **Show directions as live mockups, not prose (preferred), via a hero concept board.** For any page that matters, build a **hero concept board** per `references/concept-boards.md`: one self-contained HTML holding 6 to 10 drastically different hero variants (each a different strategic bet, not a reskin), a switcher with arrow-key nav, and a brief panel per variant (Bet / Best for / Awareness stage / CTA / Lead magnet), ordered least-aware → most-aware. Write the companion `*-brief.md` (awareness map + each concept's bet) alongside it. The user reacts by gut and locks one or two. For a small or explicitly-fast page, fall back to 2 to 3 prose directions instead (`decisions-as-options` rule: aesthetic summary + type pairing + palette + hero concept + a recommendation). **Wait for the user to lock the direction.**

Output: a hero concept board (or a "Strategy + Direction" block) the user approves.

### Phase 2: Copy (the conversion engine)

Copy carries the conversion; design serves it. Write copy before final layout so the layout fits the words. Work from `references/copywriting-playbook.md` (read it now).

Write the full deck, section by section, in priority order: **hero** (headline = the one promise, specific and benefit-led; subhead = how/for whom/proof; primary CTA label; optional eyebrow; 2 to 3 headline variants with a recommendation) → **social proof strip** → **problem/stakes** (the pain in the visitor's words) → **value/how it works** (3 to 5 outcomes, not features, each with a proof point) → **deeper proof** (named testimonials, case numbers, before/after) → **objection handling / FAQ** (the 3 to 5 real hesitations) → **risk reversal** → **final CTA** → **microcopy** (form labels, button states, privacy reassurance, footer).

- For complex offers or when the user wants variants, delegate to the **`copywriting-storytelling` agent** with offer + ICP + voice + goal + awareness stage; curate what it returns.
- Honor global writing rules: no dashes as punctuation, no dramatic fragment pairs ("Not X. Y."), complete sentences.
- **Never** use lorem ipsum, placeholder names, fake testimonials, fabricated logos, or invented stats. Missing proof gets flagged and designed around honestly.
- **Public-facing privacy.** Never name specific schools/institutions; honor any client anonymity or confidentiality rules that apply to the public copy.

Present the copy deck for approval with the structure in Phase 3.

### Phase 3: Design + structure (conversion mechanics)

Map the approved copy onto a section-by-section layout following `references/design-principles.md` (re-read the structural rules). Above the fold carries the full value proposition. One primary CTA, repeated at decision points, secondaries subordinate. Shortest viable form with reassurance microcopy. Trust seeded early, heavy proof stacked before each CTA. Scannable (F/Z pattern, real subheads, visual anchors). Mobile stack designed first. Perceived-performance budget set.

Produce a labeled section list paired with the copy and the visual treatment per section (background, type scale, motion intent, imagery). Present structure + copy + visual notes together. **Lock with the user before building.**

**Section concept boards (preferred for important pages).** Before committing to one full-page build, do for the major sections what Phase 1 did for the hero: build a **section board** per `references/concept-boards.md` for each section that carries weight (value/how-it-works, proof, pricing, the closing CTA), 2 to 4 distinct treatments each, in one switchable HTML so the user picks the strongest treatment per section by reacting to live mockups. Lock the chosen treatments, then assemble them with the locked hero into the full page in Phase 4. Skip this only for small/fast pages or when the user says "just build it."

### Phase 4: Build (assemble + elevate to full page)

Default output: a single self-contained, production-ready HTML file styled with Tailwind. Maximizes iteration speed and trivial deployment.

1. **Assemble + build with `impeccable`** as the engine: take the locked hero and the locked per-section treatments from the concept boards (or the locked structure if boards were skipped) and assemble them into one cohesive full page, feeding `impeccable` the locked strategy, direction, structure, and copy so it lands in production-grade code. Reconcile the boards into a single coherent system (one type scale, one spacing scale, one palette) rather than pasting variants that were designed in isolation. This skill retains ownership of strategy/structure/copy/conversion; `impeccable` elevates craft. Pull concrete palette/font/style from `ui-ux-pro-max` if not already locked. Add **`emil-design-eng`** for any motion.
2. **Hold the craft standard.** Read `references/craft-standards.md` and build to it: type tracking and measure, a single spacing scale, optical alignment, tinted neutrals, depth used on purpose, every interactive element with hover/active/focus, no orphans/widows. This is where good becomes world-class.
3. **Tailwind setup.** For a self-contained file, use the Tailwind Play CDN. The script tag + config syntax change between major versions, so fetch current setup via `context7` (`resolve-library-id` then `query-docs` for "Play CDN setup") before writing it. Note in the Brief that Play CDN compiles to a static build later if wanted.
4. **Fonts.** Load the chosen display + body fonts (Google Fonts or self-hosted). Preconnect + preload the hero font to avoid FOIT/CLS. Never silently fall back to a system font the design did not call for.
5. **Self-contained.** Inline critical CSS and small SVGs. Keep external requests to fonts + Tailwind runtime + necessary images. It must open correctly by double-clicking.
6. **Motion.** CSS-first. One orchestrated above-the-fold load with staggered reveals beats scattered micro-interactions. Respect `prefers-reduced-motion`.
7. **Real assets only.** Official brand logos from the project's assets folder (e.g. `<root>/Business/<Company>/Assets/`), never hand-rolled or AI-generated marks. Use the user's imagery; if a placeholder is unavoidable, label it honestly and flag it. For image *generation*, use the `image-director` skill (it writes the prompt and renders via the Codex built-in tool) rather than calling an image API.
8. **Preformatted content.** Any code/ASCII/aligned text follows `~/.claude/rules/html-authoring.md` (`white-space: pre` / `<pre>`).
9. **Forms.** Wire to the correct destination. Use the form-notification email identity that matches whichever project owns the page (per your CLAUDE.md). If a real backend is needed, ask which rather than assuming.
10. **Accessibility baseline.** Semantic landmarks, alt text, labeled inputs, visible focus, contrast ≥4.5:1 body and 3:1 large headings.

**Secondary format (Next.js):** only for an existing Next.js App Router codebase. Produce `page.tsx` + extracted components in the project's conventions, Tailwind via the project config (not CDN), fonts via `next/font`, following the project CLAUDE.md and lint rules. Files go in the project's `.Code/` tree, never repo root.

Save per the output conventions below before QA.

### Phase 5: Multi-discipline critique + visual QA loop (mandatory)

Never declare done from source. This is the gate that turns 10 correction rounds into 1.

1. **Read** `references/conversion-checklist.md` (the QA gate), `references/craft-standards.md` (the detail bar), and re-read `~/.claude/rules/completion-discipline.md` and `~/.claude/rules-ondemand/asset-production.md`.
2. **Open and screenshot** at three viewports with Playwright (`browser_navigate` to the `file://` path, then `browser_take_screenshot`; `fullPage: true` for full captures): desktop above-the-fold (1440), desktop full page, mobile (390) above-the-fold + full. Do not resize the user's Chrome; Playwright with explicit viewports is the reliable path.
3. **`Read` every screenshot and actually look.** Tool success is not verification. Walk the conversion checklist AND run the **multi-lens critique** from `references/agency-roles.md`: review the page once through each discipline's eyes (strategist, copy chief, art director, UX/CRO, motion, engineer, QA director). Each lens lists what it would reject. Catching it here is the whole point of the agency model.
4. **Critique honestly and fix.** List every failure against the checklist, the craft standard, and the lenses; fix the file; re-screenshot; re-inspect. Iterate until it passes. Offer `design-visionary` for a deeper push if the user wants.
5. **Report against the checklist.** Only "done" when every item passes. Defer anything explicitly, never silently (`completion-discipline.md`).

### Phase 6: Brief and handoff

Write a tight one-page **Landing Page Brief** (Markdown): **Goal + ICP** (one line each); **Strategy** (positioning, awareness stage, core message); **Hero angle** (chosen headline + why it beat the alternatives); **Aesthetic direction** (look, type pairing, palette); **Social proof strategy** (what proof, where, and gaps to fill); **CTA hierarchy** (primary action, where it repeats, form fields + destination); **Conversion mechanics applied**; **Open items** (missing assets, real proof to source, backend to wire, anything placeholdered); **Deploy note** (how to ship; Play-CDN-to-compiled option).

Then give the file paths and a one-line deploy suggestion.

## Output conventions and file placement

Follow the CLAUDE.md File Output Location rules exactly. Never the home root or a repo root.

- **Self-contained HTML deliverable** + **Brief** in the project's Assets area:
  - A business project: `<root>/Business/<Company>/Assets/landing-pages/<page-slug>/index.html` + `brief.md`.
  - A personal project: `<root>/Personal/<project>/Assets/landing-pages/<page-slug>/`.
- **Concept boards** (hero board, section boards, their `*-brief.md`, `image-prompts.md`, and per-concept PNG exports) in a `concepts/` subfolder of that page's folder: `.../landing-pages/<page-slug>/concepts/`. (A standalone hero exploration may also use the existing `Assets/landing-hero/` pattern.)
- **QA screenshots** in a `screenshots/` subfolder of that project's Assets, named meaningfully (`<slug>-desktop-hero.png`, `<slug>-mobile-full.png`). Delete throwaway captures; keep the final approved set.
- **Next.js code** in the project's `.Code/` tree, never repo root.

If the project folder does not exist, create the nested path. If unsure which project owns the page, ask before writing.

## Interaction with other tools and skills

- **`design-taste-frontend`**: anti-slop rulebook + pre-ship gate (Phase 1/3 direction, Phase 5 checklist). Do not override its anti-slop guidance.
- **`ui-ux-pro-max`**: concrete palette/font/style-per-product-type lookup (Phase 1/3).

- **`design-visionary` agent**: deeper design critique during QA.

- **Firecrawl**: scraping inspiration/competitor pages and category research.
- **context7 MCP**: current Tailwind / Next.js / `next/font` setup syntax at build time.

(The old `frontend-design` plugin is retired from this pipeline; `impeccable` + `design-taste-frontend` cover its visual-craft role.)
